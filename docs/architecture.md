# Архитектура решения

## Общее описание

В рамках проектной работы архитектура системы была переработана для повышения:
- отказоустойчивости;
- производительности;
- масштабируемости.

Исходное решение состояло из одного экземпляра приложения и одной MongoDB-ноды, что создавало единую точку отказа и приводило к проблемам при росте нагрузки.

В итоговом решении реализованы:
- шардирование MongoDB;
- репликация данных;
- кеширование через Redis.

---

## Компоненты системы

### Приложение
- pymongo_api — API-сервис на FastAPI

### Кеширование
- redis — используется для кеширования запросов

### MongoDB кластер

#### Router
- mongos — маршрутизирует запросы к шардам

#### Config servers (Replica Set)
- configSrv1
- configSrv2
- configSrv3

Хранят метаданные шардированного кластера:
- информация о шардах;
- распределение чанков;
- маршрутизация данных.

#### Шард 1 (Replica Set)
- shard1-1 (PRIMARY)
- shard1-2 (SECONDARY)
- shard1-3 (SECONDARY)

#### Шард 2 (Replica Set)
- shard2-1 (PRIMARY)
- shard2-2 (SECONDARY)
- shard2-3 (SECONDARY)

---

## Сетевые взаимодействия

- pymongo_api → redis
- pymongo_api → mongos
- mongos → config servers
- mongos → shard1 replica set
- mongos → shard2 replica set

---

## Шардирование

Коллекция:
somedb.helloDoc

зашардирована по ключу:
{ name: "hashed" }

Это позволяет:
- распределять данные между шардами;
- равномерно балансировать нагрузку;
- масштабировать систему горизонтально.

---

## Репликация

Каждый шард реализован как Replica Set из трёх узлов.

Это обеспечивает:
- автоматическое переключение (failover);
- сохранность данных;
- устойчивость к падению отдельных нод.

Config servers также объединены в Replica Set для отказоустойчивости метаданных кластера.

---

## Кеширование

В системе используется Redis:
- подключён через fastapi-cache2;
- используется для кеширования ответов API;
- снижает нагрузку на MongoDB.

Для эндпоинта:
GET /{collection}/users

включено кеширование с TTL 60 секунд.

---

## Изменения относительно исходной архитектуры

Было:
- одна MongoDB-нода;
- отсутствие репликации;
- отсутствие кеширования;
- прямое подключение приложения к MongoDB;
- блокирующий time.sleep.

Стало:
- шардированный MongoDB-кластер;
- Replica Set на каждом шарде;
- Replica Set config servers;
- подключение через mongos;
- Redis для кеширования;
- асинхронный asyncio.sleep.

---

## Результат

После изменений система:
- перестала иметь единую точку отказа;
- масштабируется горизонтально;
- выдерживает большую нагрузку;
- уменьшает количество запросов к базе за счёт кеша;
- быстрее отвечает на повторяющиеся запросы.
