# 7. Проектирование схем коллекций MongoDB для шардирования данных

## Цель

Спроектировать схемы коллекций `products`, `orders` и `carts`, выбрать для них шард-ключи и стратегии шардирования для:

* равномерного распределения данных;
* эффективных запросов;
* масштабируемости системы.

---

## Итоговые решения

| Коллекция | Shard key                               | Стратегия |
| --------- | --------------------------------------- | --------- |
| orders    | `{ user_id: "hashed" }`                 | hashed    |
| products  | `{ category: 1, product_id: "hashed" }` | compound  |
| carts     | `{ cart_owner: "hashed" }`              | hashed    |

---

# Коллекция orders

## Назначение

Хранит заказы пользователей.

Основные операции:

* создание заказа;
* получение истории заказов пользователя;
* получение статуса заказа.

## Схема

```json
{
  "_id": "ObjectId",
  "order_id": "ORD-2026-000001",
  "user_id": "user-123",
  "created_at": "ISODate",
  "status": "created",
  "total_amount": 159990,
  "geo_zone": "msk",
  "items": [
    {
      "product_id": "prod-1001",
      "price": 99990,
      "quantity": 1
    }
  ]
}
```

## Shard key

```js
{ user_id: "hashed" }
```

## Почему

* высокая кардинальность;
* равномерное распределение;
* быстрый доступ к заказам пользователя;
* нет hot shard, как при `created_at`.

## Индексы

```js
db.orders.createIndex({ order_id: 1 }, { unique: true })
db.orders.createIndex({ user_id: 1, created_at: -1 })
db.orders.createIndex({ status: 1 })
```

## Шардирование

```js
sh.shardCollection("somedb.orders", { user_id: "hashed" })
```

---

# Коллекция products

## Назначение

Хранит товары и остатки.

Основные операции:

* обновление остатков;
* поиск по категории;
* фильтрация по цене;
* отображение карточки товара.

## Схема

```json
{
  "_id": "ObjectId",
  "product_id": "prod-1001",
  "name": "Смартфон X",
  "category": "electronics",
  "price": 99990,
  "stock_by_geo": {
    "msk": 50,
    "ekb": 30
  },
  "attributes": {
    "color": "black"
  }
}
```

## Shard key

```js
{ category: 1, product_id: "hashed" }
```

## Почему

* поиск идёт по категории;
* `product_id` добавляет равномерность;
* нет перекоса по популярным категориям;
* баланс между каталогом и карточкой товара.

## Индексы

```js
db.products.createIndex({ product_id: 1 }, { unique: true })
db.products.createIndex({ category: 1, price: 1 })
```

## Шардирование

```js
sh.shardCollection("somedb.products", { category: 1, product_id: "hashed" })
```

---

# Коллекция carts

## Назначение

Хранит корзины пользователей и гостей.

Основные операции:

* получение активной корзины;
* изменение корзины;
* merge корзин;
* удаление по TTL.

## Схема

```json
{
  "_id": "ObjectId",
  "cart_owner": "user:user-123",
  "user_id": "user-123",
  "session_id": null,
  "status": "active",
  "items": [
    {
      "product_id": "prod-1001",
      "quantity": 1
    }
  ],
  "created_at": "ISODate",
  "updated_at": "ISODate",
  "expires_at": "ISODate"
}
```

## Shard key

```js
{ cart_owner: "hashed" }
```

## Почему

* объединяет user_id и session_id;
* все операции по корзине идут в один shard;
* высокая кардинальность;
* равномерное распределение;
* удобно для merge.

## Индексы

```js
db.carts.createIndex({ cart_owner: 1, status: 1 })
db.carts.createIndex({ user_id: 1, status: 1 })
db.carts.createIndex({ session_id: 1, status: 1 })
db.carts.createIndex({ expires_at: 1 }, { expireAfterSeconds: 0 })
```

## Шардирование

```js
sh.shardCollection("somedb.carts", { cart_owner: "hashed" })
```

---

# Общие команды

```js
sh.enableSharding("somedb")

sh.shardCollection("somedb.orders", { user_id: "hashed" })
sh.shardCollection("somedb.products", { category: 1, product_id: "hashed" })
sh.shardCollection("somedb.carts", { cart_owner: "hashed" })
```

---

# Итог

Выбранные стратегии:

* соответствуют паттернам доступа;
* предотвращают hot shard;
* обеспечивают масштабируемость;
* оптимальны для операций чтения и записи.

# Задание 8. Выявление и устранение «горячих» шардов

## Цель

Определить, как выявлять перегруженные shard'ы (hot shard) и какие меры применять для устранения дисбаланса нагрузки.

Проблема: 70% запросов приходятся на категорию `electronics`, что приводит к перегрузке одного shard.

---

# Как выявить hot shard

## Метрики мониторинга

### 1. CPU по shard

Если один shard стабильно загружен сильнее остальных → признак перегрузки.

### 2. Operations/sec

Нужно отслеживать:

* queries/sec
* updates/sec
* inserts/sec

Если один shard обрабатывает существенно больше операций — нагрузка распределена неравномерно.

### 3. Latency

* средняя latency
* p95 / p99

Рост latency на одном shard → узкое место.

### 4. Активные соединения

Если к одному shard подключений больше — он перегружен.

### 5. Распределение данных

* количество chunk'ов на shard
* объём данных на shard

Помогает понять, проблема в данных или в запросах.

### 6. Доля запросов по категориям

Если одна категория (например `electronics`) даёт большую часть трафика — это источник hot shard.

# Механизмы автоматического перераспределения

## 1. Balancer MongoDB

Переносит chunk'и между shard'ами.

```js
sh.getBalancerState()
sh.setBalancerState(true)
```

---

## 2. Добавление нового shard

```js
sh.addShard("shard3ReplSet/...")
```

Balancer автоматически перераспределит данные.

---

# Меры устранения

## 1. Изменение shard key

Плохо:

```js
{ category: 1 }
```

Хорошо:

```js
{ category: 1, _id: "hashed" }
```

👉 данные внутри категории распределяются равномерно

---

## 2. Кеширование

Использовать:

* Redis
* CDN
* API Gateway cache

👉 снижает нагрузку на MongoDB

---

## 3. Разделение данных

Вынести часто обновляемые данные (например остатки) в отдельную коллекцию.

---

# Итог

Hot shard возникает при:

* неудачном shard key;
* перекосе запросов (например одна категория).

Для решения:

* мониторим CPU, latency, ops/sec;
* используем balancer;
* при необходимости добавляем shard;
* выбираем более равномерный shard key;
* применяем кеширование.

# Задание 9. Настройка чтения с реплик и консистентность

## Цель

Определить, какие операции чтения для коллекций `products`, `orders` и `carts` можно выполнять с `secondary`, а какие должны идти только в `primary`.

---

## Таблица операций чтения

| Коллекция | Операция чтения                                     | Читать с  | Допустимая задержка репликации | Обоснование                                          |
| --------- | --------------------------------------------------- | --------- | ------------------------------ | ---------------------------------------------------- |
| products  | Чтение карточки товара                              | secondary | до 1–2 сек                     | Небольшая задержка допустима для описания товара     |
| products  | Поиск товаров по категории и цене                   | secondary | до 1–2 сек                     | Каталог может пережить небольшое отставание          |
| products  | Чтение остатков перед оформлением заказа            | primary   | 0 сек                          | Нельзя продавать отсутствующий товар                 |
| orders    | История заказов пользователя                        | secondary | до 1–3 сек                     | Небольшая задержка обычно допустима                  |
| orders    | Статус только что созданного или изменённого заказа | primary   | 0 сек                          | Пользователь ожидает актуальный статус               |
| carts     | Чтение активной корзины после изменения             | primary   | 0 сек                          | Пользователь должен видеть актуальный состав корзины |
| carts     | Чтение старых корзин для фоновых задач              | secondary | до 5 сек                       | Для фоновых процессов небольшая задержка не критична |

---

## Обоснование по коллекциям

### products

Для `products` можно читать с `secondary` операции, которые не влияют напрямую на деньги и остатки:

* карточка товара;
* каталог;
* фильтрация по категории и цене.

Но чтение остатков перед оформлением заказа должно идти только с `primary`, потому что даже небольшая задержка репликации может привести к продаже товара, которого уже нет в наличии.

---

### orders

История заказов пользователя может читаться с `secondary`, потому что небольшое отставание допустимо.

Но статус заказа сразу после оформления, отмены или изменения должен читаться только с `primary`, иначе пользователь может увидеть устаревшее состояние заказа.

---

### carts

Для `carts` активная корзина должна читаться только с `primary`, потому что корзина часто изменяется:

* добавление товара;
* удаление товара;
* merge гостевой корзины;
* оформление заказа.

Если читать активную корзину с `secondary`, пользователь может увидеть старый состав корзины.

Фоновые операции по старым или abandoned корзинам можно выполнять с `secondary`.

---

## Итог

### Можно читать с secondary

* карточка товара;
* каталог товаров;
* история заказов;
* старые корзины для фоновых задач.

### Нужно читать только с primary

* остатки товара перед оформлением заказа;
* актуальный статус заказа после изменения;
* активная корзина пользователя.

---

## Вывод

Выбор между `primary` и `secondary` зависит от риска использования устаревших данных:

* если чтение не влияет напрямую на деньги, остатки или текущий пользовательский state, допустимо использовать `secondary`;
* если устаревшие данные могут привести к ошибке оформления заказа, неверному статусу или поломанной корзине, чтение должно идти только с `primary`.


# Задание 10. Миграция на Cassandra

## Цель

Определить, какие данные стоит перенести в Cassandra, спроектировать модель данных и выбрать стратегии обеспечения консистентности при высокой нагрузке.

---

# 10.1. Какие данные переносим в Cassandra

## Подходящие данные

### 1. История заказов

* большой объём данных;
* высокая нагрузка на чтение и запись;
* хорошо подходит под модель Cassandra.

### 2. Статусы заказов

* частые обновления;
* нужен быстрый доступ по order_id.

### 3. Корзины / пользовательские сессии

* высокая частота изменений;
* важна масштабируемость;
* допустима eventual consistency.

---

## Неподходящие данные

### 1. Каталог товаров

* сложные фильтры (категория, цена);
* требует гибких запросов;
* лучше оставить в MongoDB.

### 2. Остатки товаров

* критична строгая консистентность;
* риск продажи несуществующего товара.

---

# 10.2. Модель данных Cassandra

## Таблица orders_by_user

```sql
CREATE TABLE orders_by_user (
    user_id text,
    created_at timestamp,
    order_id text,
    status text,
    total_amount decimal,
    PRIMARY KEY ((user_id), created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC);
```

### Ключи

* partition key: user_id
* clustering: created_at, order_id

### Почему

* все заказы пользователя в одной партиции;
* быстрый доступ к истории;
* хорошее распределение по пользователям.

---

## Таблица order_status

```sql
CREATE TABLE order_status (
    order_id text PRIMARY KEY,
    user_id text,
    status text,
    updated_at timestamp
);
```

### Почему

* быстрый доступ по order_id;
* минимальная задержка.

---

## Таблица cart_by_owner

```sql
CREATE TABLE cart_by_owner (
    cart_owner text,
    updated_at timestamp,
    product_id text,
    quantity int,
    status text,
    PRIMARY KEY ((cart_owner), updated_at, product_id)
);
```

### Ключи

* partition key: cart_owner
* clustering: updated_at

### Почему

* все данные корзины в одной партиции;
* быстрый доступ и обновление.

---

# 10.3. Стратегии консистентности

## 1. Hinted Handoff

Использовать для:

* корзин;
* сессий.

✔ низкая latency
✔ допустима eventual consistency

---

## 2. Read Repair

Использовать для:

* истории заказов;
* чтений, где данные часто запрашиваются.

✔ постепенное выравнивание данных

---

## 3. Anti-Entropy Repair

Использовать для:

* истории заказов;
* статусов заказов.

✔ гарантирует долгосрочную консистентность
✔ выполняется по расписанию

---

# Выбор стратегий

| Сущность        | Стратегия                  |
| --------------- | -------------------------- |
| история заказов | Read Repair + Anti-Entropy |
| статус заказа   | Read Repair + Anti-Entropy |
| корзины         | Hinted Handoff             |
| сессии          | Hinted Handoff             |

---

# Итог

## Почему Cassandra

* нет полного перераспределения данных при масштабировании;
* равномерное распределение нагрузки;
* высокая скорость записи;
* leaderless-репликация.

## Компромиссы

* eventual consistency;
* необходимость проектировать таблицы под запросы;
* сложнее поддерживать строгую консистентность.

## Вывод

Cassandra лучше использовать для:

* истории заказов;
* статусов;
* корзин и сессий.

MongoDB оставить для:

* каталога товаров;
* данных с более сложными запросами и строгой консистентностью.
